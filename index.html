Naming Convention

Use this pattern: [section-purpose]-[variant]

All lowercase, hyphens between words. No spaces. No "Frame" or "Group" in the name.

You already did this correctly for three things — and it's exactly why they work well in the pipeline:

  • Main Nav Bar — clear, descriptive, used as an instance across every page
  • Footer Component — same pattern
  • Button Component - Red Filled Default — describes what it is AND its variant state

Apply this same thinking to every section on every page.

Guide 02 — Assigned to Mayo   Updated

Components — Only What's Truly Identical

Navbar, footer, and buttons are already proper components — great. The original guide asked you to componentize every repeated section. That was over-engineered. If a section varies between pages (extra button, different subheadline, different CTA), forcing it into a single component with single-use variants defeats the purpose.

Updated approach: Claude Code does NOT need Figma components to generate code components. It reads each page section-by-section via the MCP and generates code from what it sees. What matters most is descriptive layer names (Guide 1) and consistent styles (Guide 4). Figma components are a bonus for your own workflow — not a build requirement.

The New Rule: Component vs. Named Frame

Make it a component when the element repeats with the exact same structure — same children, same layout, only text/image content changes.

Keep it as a named frame when the section varies between pages — extra buttons, different subheadlines, different element counts. A well-named frame is better than a component with one instance.

COMPONENT — identical structure, content-only overrides navbar, footer, buttons, sponsor-logo-row, social-icon-set NAMED FRAME — similar layout but small structural differences per page hero sections, two-col-text-image blocks, CTA sections, content sections with varying buttons/subheadlines DON'T DO THIS — component variant with 1 instance hero Type=Article (only used once on one page)

For Sections That Stay as Frames

The naming from Guide 1 does the heavy lifting. As long as each section frame has a descriptive name (hero-home, about-division, cta-closing) and its children are also named, Claude Code will:

  • Recognize the section's purpose from the name
  • Extract typography and colors from the applied styles
  • Generate a code component with the right structure
  • Handle page-specific variations naturally (extra button = extra slot in the component)
Bottom line: You already have the three hardest components done (navbar, footer, buttons). Add sponsors-row, stats-table, and results-table if you have time. Everything else is fine as named frames. This guide originally estimated ~3 hours — the updated scope is ~45 minutes.

Add Variants & Properties

Variants are different versions of the same component that Figma treats as one thing with toggle-able options. Claude Code reads these toggles and generates code that handles all variations automatically.

Step-by-Step: Hero Component Variants

Create the first variant

Take the Homepage hero (now named hero-home) and make it a component.

Create additional variants

Duplicate the hero component. Modify the duplicate to match the division pages hero (different layout, no video, smaller height). Duplicate again for the simple hero (FAQ, Contact).

Combine as Variants

Select all hero variants. Right-click → Combine as Variants. Figma wraps them in a dashed-border Component Set.

Name the property

Use it

On each page, place an instance of hero and toggle the Type property. No more copy-pasting separate frames.

Verify Your Design Styles

The original guide asked you to create Figma Variables from scratch. You already have color styles and text styles set up — those are exactly what Claude Code needs. Variables are optional and can't handle gradients, blend modes, or effects anyway.

What changed: The original guide said to create Figma Variables for every color. That's been replaced. Your existing color styles (Dark, White, Midtone, Accent-Bright, Accent-Dark, Glass BG Gradient) and text styles (Heading 1-4, Body Text 1-3, Button Text 1-2) are the design tokens. Claude Code reads the values from applied styles — hex codes, font specs, sizes — and we map them to Tailwind tokens on the code side.

Why Styles Work (and Why Variables Aren't Required)

When Claude Code reads a Figma node via the MCP, it extracts whatever visual properties are applied — whether those come from a style or a variable or a direct value. The difference:

  • Variables → Claude sees the token name (color/primary-red) AND the hex value
  • Styles → Claude sees the hex value directly (e.g., #E63946)
  • Either way → we define the Tailwind config on the code side and map the values to semantic token names

And critically: Variables cannot store gradients, blend modes, or effects. Figma's variable system only holds single scalar values (solid color, number, boolean). Your Glass BG Gradient style, any drop shadows, and blend modes must be styles. Don't redo what already works.

What to Do (15 Minutes)

Audit style coverage

Spot-check 3-4 pages. Are all text layers using text styles (not locally overridden)? Are all fills using color styles? Look for the small "detached" indicator in the right panel — that means a layer has a style assigned but was manually overridden.

Reattach any detached styles

If a heading uses the right font/size but isn't linked to the "Heading 1" style, select it → click the style picker → select the matching style. This ensures consistency.

Check for "raw" colors

If any section background, border, or text uses a raw hex value instead of one of your 6 color styles, apply the correct style. The goal: every color in the file should trace back to one of your defined styles.

When Would You Add Variables? (Optional / Future)

Variables become useful when you need theming (light mode / dark mode toggle) or responsive spacing tokens. For a single-theme site like Matthew Laprade Racing, your styles cover everything. If theming becomes a requirement later, you can layer variables on top of your existing styles without redoing anything.

Bottom line: Your styles are the design tokens. Don't create variables just because the old guide said to. Spot-check that styles are applied consistently across pages — that's the only task here. Estimated: ~15 minutes instead of the original 30.
